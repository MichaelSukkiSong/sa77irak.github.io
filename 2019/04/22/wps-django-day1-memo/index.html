일반적인 작업 flow:

  1. 파이썬 프로젝트 생성
  2. 가상환경 생성
  3. 장고 설치
  4. 장고 프로젝트 생성
  5. 장고 프로젝트 내부에서 할일들 진행

가상환경(virtual environment, 줄여서 virtualenv)

작업하고 있는 프로젝트의 코드를 넣을 수 있는 일종의 전용 상자.
이를 이용해 서로 다른 프로젝트마다 코드가 섞이지 않도록 유지 가능.
개발 환경을 깔끔하게 관리.
생성할 곳만 정하면 이를 만들 수 있다.

(장고걸스) 누군가가 서버에 웹사이트를 요청하면 어떤일이 벌어지는지 요약.

웹 서버에 요청이 오면 장고로 전달 됨.
장고 urlresolver는 웹 페이지의 주소를 가져와 무엇을 할지 확인.
urlresolver는 웹 사이트 주소인 URL를 통해 이해함.
장고는 패턴 목록을 가져와 URL과 맞는지 처음부터 하나씩 대조해 식별함.
만약 일치하는 패턴이 있다면 장고는 해당 요청을 관련됨 함수(view)에 넘김.
만약 데이터 수정 요청이 온다면 view함수는 수정할 수 있는 권한이 있는지 확인하고 나서 수정을 하고 답장을 주는데 장고는 그 답장을 사용자의 웹브라우저에 보내주는 역할을 함.

python3랑 python2랑 충돌해서 버츄얼 실행 안된 오류 해결

시작하기

django-admin startproject [이름]

django-admin: 장고 매니저
startproject: 프로젝트를 만들겠다라는 옵션
myside, config: 프로젝트 이름, 혹은 설정 폴더 이름
config.: 현재 디렉토리에 설정 디렉토리만 만들겠다는 의미

python manage.py [명령어]

$ python manage.py runserver [ip]:[포트번호]
내 컴퓨터에서 임시로 홈페이지를 출력할 때 쓰는 명령
소스코드 변경이 자동으로 반영되나 폴더나 파일을 추가 생성한 경우는 자동반영이 불가.
포트번호 생략시 기본 8000
ip 생략시 기본 127.0.0.1

만들어지는 파일들에 대해

venv 디렉토리:
가상환경 생성시 만들어지는 디렉토리

manage.py:
사이트 관리를 도와주는 역할

프로젝트 이름명을 가진 디렉토리 내의 파일들 설명

settings.py:
웹사이트 설정이 있는 파일
설정값을 저장하는 파일.
크리티컬한 정보는 운영체제 환경변수 값을 사용하거나, 별도 파일에서 불러온다.

urls.py:
urlresolver가 사용하는 패턴 목록을 포함.
주소 설계 내용을 저장하는 곳. (파일을 여러개 나누는 것을 분기를 나눈다고 한다. 이를 라우팅한다고 한다.)

wsgi.py:
웹 서버: Apache, NginX- 웹 서비스의 집사, 캐셔, 안내데스크 역할을 한다.
Tomcat
CGI →FastCGI → mod??? : 맥도날드에서 주문이 들어오면 켜서가 주문이 들어온만큼 자고 있는 크루를 깨워서 햄버거를 만들게 시킴. 이러면 깨우는 시간, 주문이 많으면 크루가 필요하여 효율이 좋지 않음.
Middleware → WSGI: Gunicorn, UWSGI(이 파일을 메모리에 항상 상주 시킴)
웹 어플리케이션: 메모리 상주(장고)
웹 어플리케이션 서버: 장고 같은 스크립트로 작성된 어플리케이션

앱

장고에서 앱은 재사용을 위해서 만드는 기능 단위를 말한다.
어떤 페이지나 기능을 만들기 위해서는 앱을 만들고 그 안에 만든다.

python manage.py startapp [앱이름]

앱을 만들면 앱 이름으로 폴더가 생성됨

앱 폴더안에 있는 파일들

migrations 폴더 :
DB 변경 사항 이력 파일들. 이전 상태로 돌아가거나 하는 것이 가능하다.

admin.py :
모델을 관리하기 위해 등록, 관리자 페이지 커스터마이징.

apps.py :
앱 정보 관리, 앱 진입시 로드할 기능 관리. 수정할 일이 거의 없음.

models.py :
모델을 만들고 관리하는 부분
모델(Class)은 데이터베이스에 어떤 자료를 어떤 형식으로 저장할 것이냐는 것을 지정하는 것.
ORM : Object Relational Mapping : 데이터를 객체에 맵핑하는 것. 쿼리문을 ORM이 자동으로
사용하는 데이터베이스에 맞게 알아서 변경해줌.

tests.py :
테스트 셋을 지정하는 파일 - TDD. 공부할 필요성은 있으나 필수는 아님

views.py :
페이지나 기능을 코딩하는 파일

  1. 매뉴얼적인 부분
    1-1. 튜토리얼적인 부분
  2. 관례적인 부분
  3. 실무적인 부분

오류

문법 오류 : 문법적 문제, 오타 80% -> 해결볍 : 자동완성 사용
런타임 오류 : 실행시점 오류
논리 오류 : 수치를 0으로 나누는 경우. 회의에서 근거가 없는 경우
