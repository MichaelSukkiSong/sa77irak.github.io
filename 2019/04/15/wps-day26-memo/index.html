#데이터베이스 1일차 녹음 (3:22:00~)

mysql -u root
일케 하면 데이터베이스에 접속하지.connection 연결 . 데이터의 무결성이 보장되어야 하니 TCP connection~소켓을 이용해서 TCP를 뚷은거. 데이터베이스에서는 TCP를 뚫은거를 session을 열었다 함. 즉 TCP 연결을 했다 의미~

mysql server ->이게 유투브인셈 <---tcp 커넥션--->client →mysql(database client)->이게 크롬

mysql john mysql greg
client client

server에서는 listen socket이 필요하고 각각의 client에 대해서 쓰레드가 따로 돌면서 data socket이랑 연결되겠지. 데이터베이스는 안에 테이블을 가지고 있지. 공유자원에 대해서 여러개의 쓰레드가 동시에 쓰면 race condition! table이 shared recource 즉 공유자원이 됨. race condition! 개념이 있겠지.

만약 어떤 한 클라이언트가 update문을 통해서 balance를 5천원에서 3천원으로 떨어뜨리고 상대방한테 전송중인데 접속이 끊겨버림!. 상대방에 대해서 2천원을 올리기전! 그럼 문제 겠지! 그래서 디비는 무결성을 보장해야함..! 매번 반영되버리면 난리나는거지 그렇기 때문에..

*데이터베이스 무결성(transaction과 session의 차이, 즉 session은 tcp connection의 의미고 transaction은 DB 처리 작업이 모두 왼료되거나 아니면 모두 취소되게 되는 작업의 단위를 의미. 즉 transaction은 업데이트 되는 단위를 의미 ):
1)DML의 경우: COMMIT 하기 전까지는 transaction이 완료 안됨. 즉 다른 client한테까지 반영 안됨.(DML의 경우를 말하는 것, UPDATE/DELETE 명령 같은 경우). 즉 DML같이 데이터 조작을 하는 경우에는, 업데이트를 하는 순간 그 쓰레드에 대해서만 바뀌고 나머지 애들은 그대로 있고 실제 테이블 값도 update했다고 바뀌지는 않음. 이 업데이트 단위를 transaction이라 함. update한 후에 commit이라는 명령어를 통해 transaction을 끝내야 함. 실제 데이터베이스의 값들을 commit한 순간 바뀜. 그래야 나머지 클라들도 따라서 다 바뀜. commit전에는 어떤 dml 작업을 하더라도 절대 나머지 디비애들한테는 영향을 미치지 않는다는 데이터베이스 무결성 대전제임.
→ROLLBACK 명령: commit하기 전에 무효화 하겠다.

2)DDL의 경우 바로 됨. 즉 바뢰 commit 됨. 다른 client에서도 자동 업데이트 되서 공유됨. 자동으로 transaction이 닫힘. DDL같은 경우는 자동 commit임. drop같은 경우는 복구가 안됨.

commit을 하는 순간 lock이 걸림~.
