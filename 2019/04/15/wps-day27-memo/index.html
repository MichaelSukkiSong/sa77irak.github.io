###데이터베이스 2일차

#join

foreign key : table간의 관계 형성할 때 잡지.

1:N 관계, rooms의 bid가 buildings의 bid를 레퍼런스 할 수 있도록 할 때. 부모와 자식 관계.

Join 한다음에,
M:N 관계 (장고에서도 내부적으로 테이블을 M:N으로 설정하는 개념이 있음.

Join:
Inner join
Outer join: Left outer, Right outer, Full outer

mysql이나 mariaDB는 full outer join을 지원하지 않음.
하지만 UNION문 을 통해 해볼거임.

schema: 테이블이 어떤 모양으로 만들건가의 의미임.
schema를 만들 때 데이터의 중복을 허용하지 않는다가 전제임. 그래서 테이블을 쪼갬.

join은 table을 가로로 연결할 때 쓰임!!
일반적으로 join하면 inner join을 함.

#GROUP BY HAVING

WHERE문: grouping 전에 조건절 수행
HAVING문: grouping 후에 조건절 수행

HAVING은 반드시 그루핑 한 것에 대해서만 조건을 먹이는 것! 즉 group by에 대한 조건문임. Group by가 없으면 안쓰겠지.

M:N 관계 → 학생과 수강과목의 관계처럼. 서로가 다 대 다인 관계.

Primary key 안써도 unique not null 쓰면 완벽하게 대체 됨.

#view = select문임. 마치 table처럼 쓰는 것.
1.select문일 뿐이지 table이 새로 만들어지는 것은 아님.
2.가독성이 높아짐.
3.보안 문제 때문에 씀!. view를 통해 다른 민감한 정보를 못보게 뽑아내서 유저한테 grant하게 해서 보안 높아짐.

#index
index는 데이터 써치(검색)에 씀.

Tree->BST → O(logn)

Index는 BTree로 만들어짐.

BST를 보완한게 균형이진 트리(그중 하나가 red-black tree) 이고 한번 더 업그레이드 한게 B-tree

index 종류:
*clustered index:
1)한 table마다 1개씩만 존재
2)data자체를 이 clustered index에 맞추어서 저장.
3)primary key가 clustered index가 되는 것. Create table할 때 primary key를 걸어두고 작업하는게 좋지 나중에 데이터 들어간 후 나중에 정렬하려면 하루 걸릴 수도 있음 ㅜㅜ.

*secondary index:
1)우리가 원하는만큼 붙일 수 있음.
2)B-tree라는 자료구조를 내부적으로 만들고, 얘가 실제 데이터를 참조하는 식으로 되어있음.
3)where를 자주 쓰는 상황에서 secondary index를 쓰면 됨. where절에 나오는 field name에다가 index를 걸어주면 됨.

index는 어떤 field를 기준으로 자료를 검색.field에다가 index를 붙임.
index는 where에 걸리는것. Where classID=3 이런것 처럼.
index의 장점은 검색속도 빨라짐.
단점은 insert나 update문 등의 데이터 수정 삽입을 많이 일어나면 성능이 현저히 떨어짐.

1)where를 얼마나 많이 쓸건가
2)검색속도와 vs 수정/삽입해서의 성능
—-→ 를 고려해서 index를 만들지 말지 판단해야~

CREATE INDEX <–이 명렁어는 secondary index만 만들 수 있다.

Clustered index를 만들려면 만드시 ALTER TABLE문을 이용해야 함.
