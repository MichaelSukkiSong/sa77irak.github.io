index

index는 데이터 써치(검색)에 씀.

Tree->BST → O(logn)

Index는 BTree로 만들어짐.

BST를 보완한게 균형이진 트리(그중 하나가 red-black tree) 이고 한번 더 업그레이드 한게 B-tree

index 종류:
*clustered index:
1)한 table마다 1개씩만 존재
2)data자체를 이 clustered index에 맞추어서 저장.
3)primary key가 clustered index가 되는 것. Create table할 때 primary key를 걸어두고 작업하는게 좋지 나중에 데이터 들어간 후 나중에 정렬하려면 하루 걸릴 수도 있음 ㅜㅜ.

*secondary index:
1)우리가 원하는만큼 붙일 수 있음.
2)B-tree라는 자료구조를 내부적으로 만들고, 얘가 실제 데이터를 참조하는 식으로 되어있음.
3)where를 자주 쓰는 상황에서 secondary index를 쓰면 됨. where절에 나오는 field name에다가 index를 걸어주면 됨.

index는 어떤 field를 기준으로 자료를 검색.field에다가 index를 붙임.
index는 where에 걸리는것. Where classID=3 이런것 처럼.
index의 장점은 검색속도 빨라짐.
단점은 insert나 update문 등의 데이터 수정 삽입을 많이 일어나면 성능이 현저히 떨어짐.

1)where를 얼마나 많이 쓸건가
2)검색속도와 vs 수정/삽입해서의 성능
—-→ 를 고려해서 index를 만들지 말지 판단해야~

CREATE INDEX <–이 명렁어는 secondary index만 만들 수 있다.

Clustered index를 만들려면 만드시 ALTER TABLE문을 이용해야 함.

*데이터베이스 무결성(transaction과 session의 차이, 즉 session은 tcp connection의 의미고 transaction은 DB 처리 작업이 모두 왼료되거나 아니면 모두 취소되게 되는 작업의 단위를 의미. 즉 transaction은 업데이트 되는 단위를 의미 ):
1)DML의 경우: COMMIT 하기 전까지는 transaction이 완료 안됨. 즉 다른 client한테까지 반영 안됨.(DML의 경우를 말하는 것, UPDATE/DELETE 명령 같은 경우). 즉 DML같이 데이터 조작을 하는 경우에는, 업데이트를 하는 순간 그 쓰레드에 대해서만 바뀌고 나머지 애들은 그대로 있고 실제 테이블 값도 update했다고 바뀌지는 않음. 이 업데이트 단위를 transaction이라 함. update한 후에 commit이라는 명령어를 통해 transaction을 끝내야 함. 실제 데이터베이스의 값들을 commit한 순간 바뀜. 그래야 나머지 클라들도 따라서 다 바뀜. commit전에는 어떤 dml 작업을 하더라도 절대 나머지 디비애들한테는 영향을 미치지 않는다는 데이터베이스 무결성 대전제임.
→ROLLBACK 명령: commit하기 전에 무효화 하겠다.

2)DDL의 경우 바로 됨. 즉 바뢰 commit 됨. 다른 client에서도 자동 업데이트 되서 공유됨. 자동으로 transaction이 닫힘. DDL같은 경우는 자동 commit임. drop같은 경우는 복구가 안됨.

Three-way handshaking

Host 1 host 2
send buffer send buffer
recv buffer recv buffer

send buffer recv buffer 는 커널이 관리함.
소켓마다 send buffer와 recv buffer가 따로 있다.(그냥 이리 알아둬라)

unicode문자열을 받으면 encode해서 b’’이런식이 되지. 이게 byte string이지. byte단위로 쏘는 스트링을 byte string b’\ \ \’여기서 하나씩 쏘는거. b’abc’라면 host1의 send buffer에 ‘abc’됨. 그래서 결국 host2의 recv buffer에 ‘abc’가 되는것이 목표지. 그래서 받고 나서 decode하는 과정임. Send buffer에 첫번째 바이트에 부여한 숫자를 ISN(Initial sequence number)이라 함. 예를 들어 1023 같이 ISN 지정. 이 다음은 1024였지만 해커 땜에 랜덤하게 부여됨.
host2도 받으면 쏴야하니 send buffer의 ISN는 62523으로 됬데.

1)host1이 연결하자 ~ 나랑 동기화할래? SYN을 보냄~ 그리고 내 seq#는 1023이야~

2)host2는 아 그래?내 recv buffer의 첫번째 바이트를 1023으로 초기회 해놓고(이게 동기화,host1의 send buffer는 host2의 recv buffer랑 동기화되겠지.)
host2도 보낼 수 있어야하니 send buffer의 ISN을 62523 해놓고 SYN+ACK(ackoledgement) seq#=62523 ack#=1024 보냄. 즉 host1의 recv buffer의 ISN을 초기화 시키는 정보도 주고 1023 다음인 1024도 전달하는거지.

3)마지막으로 host1은 ack하면서 ack#=62524 를 보냄.

이게 three-way handshaking. 이 때 데이터를 주는건 아님. 데이터 주기 전 전초 작업 같은 것.
